Models that won't explain themselves

Modern machine learning models — gradient boosted trees, neural networks, ensembles — can be remarkably accurate. They can also be completely opaque. A model can deny someone a loan, flag a transaction as fraud, or predict a disease risk, and offer nothing more than a single number.

That's a problem for trust (would you act on a number you can't question?), for debugging (is the model picking up a real signal, or a weird artifact in the data?), and often for regulation (many industries legally require a reason for a decision).

SHAP doesn't change what the model predicts. It adds a second model on top — an explanation layer — that breaks any single prediction down into the contribution of each input feature. Tap the cards below to see the difference.

The promise

One framework, any model

SHAP works the same way whether the underlying model is a decision tree, a random forest, an XGBoost ensemble, or a deep neural network — it treats the model as a black box and only looks at its inputs and outputs.

Tap to flip ↻
The catch

It explains decisions, not the model

SHAP tells you why this prediction came out the way it did. It doesn't rewrite the model into something simpler — the original model is still doing the work. Think of SHAP as an X-ray, not a replacement skeleton.

Where the math comes from

Splitting the credit, fairly

SHAP is built on Shapley values, an idea from cooperative game theory introduced by Lloyd Shapley in 1951 — decades before anyone used it on machine learning models. The original question had nothing to do with AI:

"A group of people work together and earn a payoff together. Some combinations of people are more productive than others. How do you split the total fairly, given everyone's contribution?"

A SHAP value is a Shapley value — where the "players" are your model's input features, and the "payoff" is the model's prediction. Next, let's see exactly how those numbers get calculated.

Under the hood

How a SHAP value is actually computed

For one feature i, its SHAP value φᵢ is the average marginal contribution of that feature across every possible order in which features could be "added" to the model.

φᵢ =  |S|! · (|F|−|S|−1)!|F|!  ×  [ f(S ∪ {i}) − f(S) ]   — averaged over every subset S of features that excludes i
F = the full set of features · S = a subset that doesn't include feature i · f(S) = model's prediction using only features in S
f(S) — what the model predicts when only the features in group S are "known" (the rest are averaged out)
f(S∪{i}) − f(S) — the "marginal contribution": how much the prediction changes when feature i joins group S
the fraction — a weight that makes sure every possible ordering of features counts equally

That formula looks intimidating, but it boils down to one sentence: "Imagine features arriving one at a time, in every possible order. Average how much each feature changes the prediction when it arrives." The lab below works through a real example, step by step.

Speaking the language

Four plots you'll see everywhere

SHAP values are just numbers — but most of their power comes from how they're visualized. Here's how to read the four most common SHAP charts. Note: classic SHAP plots usually use red/blue. To stay consistent with this guide's palette, low feature values are shown in cyan and high values in pink.

Waterfall plot — anatomy of one prediction

Reads top to bottom (or left to right): start at the base value — what the model predicts on average, with no information about this particular case. Each bar then adds or subtracts, in order of importance, until you land exactly on the final prediction.

How to read it: bars pointing one way (here, cyan/green) push the prediction up; bars pointing the other way (pink) pull it down. The length of each bar is its SHAP value. The final position is the model's actual output for this one case.

Force plot — the same story, compressed

A force plot squeezes the waterfall onto a single line. Features pushing the prediction higher are bundled on one side, features pushing it lower on the other — like two teams in a tug-of-war, with the rope's resting point as the base value and the final knot position as the prediction.

How to read it: the size of each segment shows how much "force" that feature applies. It's compact and great for comparing many predictions side by side, but harder to read exact values than a waterfall.

Summary (beeswarm) plot — the whole dataset at once

Instead of one prediction, this shows every prediction in your dataset, one dot per data point. Each row is a feature, ordered by overall importance (most important at top). A dot's horizontal position is its SHAP value for that data point; its color shows whether that feature's value was high (pink) or low (cyan) for that point.

How to read it: if pink dots cluster on the right and cyan dots cluster on the left for a feature, that means high values of that feature increase the prediction — and vice versa. Wide spread = the feature matters a lot, in either direction.

Dependence plot — one feature, in detail

This plots a single feature's actual value (x-axis) against its SHAP value (y-axis) across the dataset. It reveals the shape of the relationship — is it a straight line? A curve? A threshold where things suddenly change?

How to read it: here, "Credit score" has a roughly increasing relationship with its SHAP contribution, but it flattens out — past a certain score, a higher score barely helps any more. Patterns like this are hard to spot from raw data, but jump out immediately in a dependence plot.

A common mix-up

"Most important" isn't one thing

Before SHAP, the standard way to ask "which features matter?" was a model's built-in feature importance — for tree models, often gain or split count; for linear models, the size of a coefficient. These give one global ranking, baked into the model itself. SHAP gives you something related but different: it can produce that same kind of global ranking, and it can tell you what mattered for one specific prediction. Mixing these two up is one of the most common beginner confusions.

Built-in importance

"How much did this feature help, on average, across all training data?"

Computed once from the model's internals (e.g. how often a tree splits on this feature, and how much that split reduced error). It's a single global score — it has no idea what your specific loan applicant looks like.

Mean |SHAP value|

"How much did this feature move predictions, on average, across all data?"

Also a global ranking — but built by averaging the absolute SHAP value of each feature across every prediction. Notice the order can shift: SHAP measures actual impact on outputs, not internal model mechanics.

Why don't they match? Built-in importance asks "how much does the model rely on this feature structurally" — a feature can get many splits without moving predictions much. Mean |SHAP value| asks "how much does this feature actually change the output, on average" — which is usually what you actually want to know. They often agree on the top feature or two, then diverge.
